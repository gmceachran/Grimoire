// fs (File System): Analogy: is like the building's internal mail and document handling system.
// It is a giant and intricate series of chutes that connect to all the offices (route handlers) 
// that mail flies through. Presumably there is a large mail room somewhere, possibly adjacent to the database.
// It knows how to access all files and folders in the building
// It can evaluate and manipulate all documents

// Express is the blueprint the Grimoire Data building (app) is built off of.
// App: is the Grimoire data building itself

// Path: a routing device 
// .join: a method of the roouting device that finds the easiest way from your location to another 

// Port 3000: THis is the address of the Grimiore Data Building. It's how visitors know where to go. 

// __dirname: a node variable that gives the absolute path to the file's current directory.
// E.G. tells the http request (visitor from the browser building) where it currently is.
// It would be like having a GPS or a directory with a "you are here" note.

// Simple analogous example of route handlers and middleware:
// .use(): Registers middleware: How to remember this?

// Middleware is like the receptionist between the front door and your office
// The front door is the entrance to the server
// The HTTP request is the person trying to enter the office through the front door. 
// The middleware (receptionist) validates the HTTP request by checking if it is allowed in, or it can modify the request
// (like adding a visitor badge), etc...
// The route path would be like an office directory, it explains where visitors can go. 
// The route handler is the office itself, where the actual work happens. This is where the CRUD operations actually fire.

// Middleware has two jobs here, a direcctory that shows requests how to get to the public room
// a Decoder that decodes JSON

// Why is static at the very front? 
// Static files are the most common requests, it's more efficient to move what they need closer to the entrance of the building
// Security: the public folder is like an organized document room. 
// Only safe pre-approved files are stored here. 
// It's separate from sensitive areas, like the database.

// JSON sent from building one (browser) is like an encoded message
// The express.json() receptionist's job is to decode the message before it gets sent elsewhere.
// So that the JSON can be used and organized in the accepted language of the server

// DATA_FILE: This is like printing a directory that shows how to get to grimoire-data, for future use
// Used by other parts of the office, when they need to read from, write to, 
// and update the database
// Unlike express.static (an active/updating directory for visitors
// (incoming http requests from the browser)), this is a permanent directory.

// The next section is the route handlers, or the actual offices.
// The offices are divided into departments. The first department is the chapter department.
// The chapter department is divided into four subdepartments based off of the basic CRUD 
// operations: get, post, put, and delete. 
// The get subdept has a chalkboard with all the chapters and data scrawled all over it.
// The post subdept has a post office little glass box thing that the person sits behind, he 
// puts a stamp on every new chapter.
// The delete subdept has a giant wall trash hatch thing that you open and throw the old data into.
// When you do, the lights blare and a trash alarm sounds. 
// The put subdept has a massively oversized typewriter with scrolls of paper feeding into it and out of it
// and a keyboard with hundreds of keys and even pedals like an organ. Very steampunk. The guy playing it is 
// making a clanking clacking steampunky sound instead of music.

// Note on the visitor, this has been notated elsewhere, but I'd like to assemble it here:
// The visitor is an HTTP request. It shows up with a briefcase containing credentials and information 
// about where it needs to go, what it wants to do, and what data it wants to do it to, and a response form.
// It also has a classy fedora.
// The where, what, and who info is all in a folder labeled `req`. The response form is in a folder labeled `res`. 
// The visitor is first checked in security (implicit middleware), submits his req file for JSON decoding, 
// and is then given a directory to get to its specific route handler in the server.

// 📚 GET /api/chapters
// This office is in the read/get subdept. 
// Its job is to return a list of all the available chapters currently in the database
// After the visitor (HTTP request) passes through reception (middleware), they’re directed to the Chapters Department.
// They’re carrying a briefcase. Inside are a few things, a packet labeled `req` of credentials and instructions for what they need,
// an empty form labeled `res`—which needs to be filled out by this office before they leave,
// and last, a small encoder device they can use later to store the data properly.
//
// The Chapters Department has one job: respond to chapter info requests.
// To do this, they ask the FS Team (fs.readFileSync) to fetch a document from the internal archive.
// They hand the FS Team the exact location (DATA_FILE), so FS knows where to go.
//
// The FS Team walks into the giant archive library and pulls a file—rows of punchcards (raw bytes).
// They use their UTF-8 translation book to decode it into a long scroll of JSON text,
// then hand it back to the Chapters Department.
//
// The department runs this scroll through their JSON.parse smart printer, which transforms the text
// into a clean, labeled JavaScript object—the kind of organized data they can actually work with.
// They label this packet `data`.
//
// From this, chapters get extracts just the `chapters` section and hand it over to the visitor.
// The visitor then uses their encoder to scan this section and fill out their `res` form, with the proper JSON syntax.
//
// Now that the form is complete, the visitor can carry it back across the street to the Browser Building.

// 📨 POST /api/chapters
// This office is in the post/create subdept. 
// Its job is to add new chapters to the database.

// Just like in the first office, a visitor (http request) shows up with a briefcase, 
// in which are two packets labeled req and res. The req folder, in this case, includes new data that was 
// added by the user in the browser. This data was translated at the JSON decoding desk in reception, and is 
// now accessible to the post office. 

// The visitor shows up at the glass box and hands the req and res folders over. 
// The first thing the clerk does is make a copy of a particular page in the req folder, the name page,
// on the copy machine to his right, and saves the copy for later. 

// The clerk has a monocle that he uses to check if the page is empty or not a string:
// a boolean or number. 
// If it is, he takes out an HTTP Status code binder and gets one of the 400 forms 
// (server couldn't understand request). He then takes a giant red stamp, and stamps the 
// response form with 'Invalid chapter name.' And staples the 400 sheet to it, 
// and hands it back to the visitor to return to the browser building. 

// If the name passes the initial inspection, the clerk turns behind him and pulls a lever next to the fs mail system hatch. 
// Requesting, like above, grimoire data (accessible via the data file directory).
// The fs team returns the appropriate manuscript through the fs chutes, translated from the punchcard archives using UTF-8,
// The clerk then takes the manuscript to the JSON smart printer and gets his manuscript, neatly organized into a JS object,
// He stamps this one 'data'

// The clerk returns to his desk and, using his monocle, carefully inspects his new data manuscript to see if there are 
// any chapters that already have the name. If there are, he opens his HTTP status code binder, takes out the 409 form 
// (The request conflicts with the current state (Duplicates)), takes out a giant stamp that says 'Chapter already exists',
// and stamps the res form, before stapling the 409 form to it, handing it back to the visitor to return to the browser building.

// The clerk must now create a new file with the necessary information for the fs encryption team to code and send to 
// the punchcard archives. On the back wall, next to the fs chute, is a Hollerith machine. The clerk opens the data manuscript, 
// removes the page chapter table of contents, and inserts it into the machine. This returns a card stamped with ch(#).
// He next takes his name page to a typewriter, inserts it, and gets the original and a new page titled 'slug' with the same name 
// in lower case and hyphenated where there were spaces. He takes all three papers — name, ID card, and slug — puts them in a folder, 
// and stamps that folder with a newChapter stamp.

// The clerk then creates a copy of the newChapter folder. 

// The original is then added to the back of the end of the chapters section of the data manuscript.

// The clerk then stamps the data manuscript with 'null, 2'. This is then placed in the fs chute. The clerk closes the hatch and pulls the lever, 
// which sends it to the fs team. FS uses their own smart printer (JSON.stringify) to recode the JS object (data) to JSON, now including the
// new chapter as one giant string.
// Null: (sometimes the fs team is required to attach specific addons (callback functions) to the printer to encode the data in a particular way, 
// this just a stand-in that tells them not to do that.
// 2: notates formatting, 2 spaces for each level of nesting. 

// The second newChapter folder is then used to fill out the response form. The clerk is careful to do this in JSON, so the visitor can return
// with a JSON string instead of a JS object. He then takes out his HTTP handbook, takes out a 201 card (Successfully Created) and 
// staples it to the form, and hands it back to the visitor.
